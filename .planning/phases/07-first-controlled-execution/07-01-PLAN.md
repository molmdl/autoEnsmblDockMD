  <name>Task 1: Create dryrun report generator script</name>
  <files>scripts/phase7/07-dryrun-report.sh</files>
  <action>
Create a bash script `scripts/phase7/07-dryrun-report.sh` that generates a comprehensive markdown dryrun report. The script should:

1. Accept parameters:
   - `--config <path>`: Config file path (required)
   - `--mode <blind|targeted>`: Docking mode (required)
   - `--output <path>`: Output report path (default: `.planning/phases/07-first-controlled-execution/07-dryrun-report-{mode}.md`)

2. Execute these checks and capture results:
   a. Preflight validation: Call `scripts/infra/plugins/preflight.py` with the config
   b. Tool availability: Parse preflight output for tool check results
   c. File readiness: Check existence of required inputs from config
   d. Dry-run commands: Run `scripts/run_pipeline.sh --config <config> --dry-run` and capture command list

3. Generate markdown report with sections:
   - **Summary**: Pass/fail status for each check category
   - **File/Config Readiness**: Table showing file existence and config section completeness
   - **Tool Availability**: Table showing gmx, gnina, gmx_MMPBSA availability
   - **Command Preview**: Full list of commands that would execute (collapsed in `<details>`)
   - **Workflow Flowchart**: Call the flowchart generator (Task 2) and embed output
   - **Manual Approval Gate**: Clear indicator that human review is required before real execution

4. Use existing project conventions:
   - Source `scripts/setenv.sh` at the start
   - Use `log_info`, `log_warn`, `log_error` from `scripts/infra/common.sh`
   - Use `jq` for JSON parsing (preflight output)
   - Follow error handling patterns from other wrapper scripts

5. Exit codes:
   - 0: All checks passed (ready for execution)
   - 1: Critical failure (blocking issues found)
   - 2: Warnings present (needs review but can proceed)

DO NOT modify existing pipeline scripts - only read their outputs.
DO NOT duplicate preflight logic - call the existing plugin.
</action>

  <files>scripts/phase7/07-generate-flowchart.py</files>
  <action>
Create a Python script `scripts/phase7/07-generate-flowchart.py` that generates ASCII art flowchart of the workflow stages. The script should:

1. Accept parameters:
   - `--mode <blind|targeted>`: Docking mode (required)
   - `--output <path>`: Output file path (default: stdout)

2. Generate flowchart showing:
   - Stage sequence (Stages 0-6): preflight, rec_prep, rec_prod, rec_ana, rec_cluster, rec_align, dock_prep, dock_run, dock2com, com_prep, com_prod, com_mmpbsa, com_ana
   - Sub-steps within each stage (from `scripts/run_pipeline.sh` stage registry)
   - Scripts called at each sub-step (from STAGE_CMD array)
   - Mode-specific branches (targeted mode shows reference_ligand branch)

3. Use ASCII art format:
   ```
   ┌─────────────────────────────────────────────────┐
   │ WORKFLOW FLOWCHART: Targeted Docking (Mode A)  │
   ├─────────────────────────────────────────────────┤
   │                                                  │
   │  Stage 0: Preflight Validation                   │
   │     └── scripts/infra/plugins/preflight.py      │
   │              │                                   │
   │              ▼                                   │
   │  Stage 1: Receptor Preparation                   │
   │     ├── scripts/rec/0_prep.sh                   │
   │     ├── scripts/rec/1_pr_rec.sh                 │
   │     ├── scripts/rec/3_ana.sh                    │
   │     ├── scripts/rec/4_cluster.sh                │
   │     └── scripts/rec/5_align.py                  │
   │              │                                   │
   │              ▼                                   │
   │  Stage 2: Targeted Docking                       │
   │     ├── scripts/dock/1_rec4dock.sh              │
   │     ├── scripts/dock/2_gnina.sh                 │
   │     ├── scripts/dock/3_dock_report.sh           │
   │     └── scripts/dock/4_dock2com_2.py            │
   │              │                                   │
   │  [... continue for all stages ...]              │
   │                                                  │
   └─────────────────────────────────────────────────┘
   ```

4. Extract stage information from:
   - `scripts/run_pipeline.sh` arrays: ALL_STAGES, STAGE_DESC, STAGE_CMD
   - Mode-specific behavior from config file (if provided via --config)

5. Handle both modes:
   - Blind mode: Show generic docking flow

DO NOT hardcode stage sequences - parse from run_pipeline.sh.
DO NOT create overly wide flowcharts - wrap at 80 characters.
