### Pattern 1: Safe SBATCH Script Generation

**What:** Use quoted heredocs with placeholder substitution to prevent shell injection in dynamically generated batch scripts.

**When to use:** Any time bash variables are interpolated into SBATCH scripts or other shell code generation.

**Example:**
```bash
# UNSAFE (current pattern in 1_pr_prod.sh:244-275)
cat > "${job_script}" <<EOF
#!/usr/bin/env bash
#SBATCH -J ${lig_name}_eq_${trial_idx}
cd ${q_lig_dir}
EOF

# SAFE (recommended pattern)
cat > "${job_script}" <<'EOF'
#!/usr/bin/env bash
#SBATCH -J __JOB_NAME__
cd __LIG_DIR__
EOF
sed -i "s|__JOB_NAME__|${lig_name}_eq_${trial_idx}|g" "${job_script}"
sed -i "s|__LIG_DIR__|${q_lig_dir}|g" "${job_script}"
```

**Rationale:** Single-quoted heredocs (`<<'EOF'`) prevent variable expansion during heredoc creation. Subsequent sed substitution ensures proper escaping because sed operates on literal strings, not shell metacharacters.

**Source:** ShellCheck SC2086, OWASP Command Injection Prevention Cheat Sheet

### Pattern 2: Defensive File Operations in Python

**What:** Always use `missing_ok=True` for `unlink()` operations and wrap file operations in try-except blocks.

**When to use:** Any destructive file operation, especially in cleanup/cache management code.

**Example:**
```python
# UNSAFE (current pattern in conversion_cache.py:78)
for path in self.cache_dir.glob("*"):
    if path.is_file():
        path.unlink()  # ← Raises FileNotFoundError if concurrent deletion

# SAFE (recommended pattern)
for path in self.cache_dir.glob("*"):
    if path.is_file():
        try:
            path.unlink(missing_ok=True)  # Python 3.8+
        except OSError as e:
            logger.warning(f"Failed to delete {path}: {e}")
            continue
```

**Rationale:** File operations in HPC environments may be subject to:
- Concurrent access from parallel jobs
- NFS delays/inconsistencies
- Quota-related failures

Using `missing_ok=True` and exception handling prevents workflow crashes from transient filesystem issues.

**Source:** Python pathlib documentation, Python 3.8+ Path.unlink() API

### Pattern 3: Retry Logic for Scheduler Operations

**What:** Implement exponential backoff retry logic for transient scheduler states instead of failing immediately.

**When to use:** Slurm job status checking, any operation dependent on scheduler state propagation.

**Example:**
```python
# UNSAFE (current pattern in executor.py:399-407)
if status == 'UNKNOWN':
    terminal_state = self.get_job_terminal_state(job_id)
    if terminal_state in self.FAILURE_STATES:
        return (terminal_state, output_file)
    return ('FAILED', output_file)  # ← Assumes failure for ambiguous state

# SAFE (recommended pattern with retry)
if status == 'UNKNOWN':
    # Retry with backoff for transient scheduler state
    max_retries = 3
    for attempt in range(max_retries):
        time.sleep(poll_interval * (2 ** attempt))  # Exponential backoff
        terminal_state = self.get_job_terminal_state(job_id)
        if terminal_state != 'UNKNOWN':
            return (terminal_state, output_file)
    
    # Only fail after exhausting retries
    logger.error(f"Job {job_id} in indeterminate state after {max_retries} retries")
    return ('FAILED', output_file)
```

**Rationale:** Slurm status propagation can have race conditions:
- Job completes between `squeue` and `sacct` queries
- `sacct` database replication lag
- Short-lived jobs disappearing from queue before status check

Retry with backoff gives scheduler time to propagate state consistently.

**Source:** Google SRE Book - Retry Logic Best Practices, Slurm documentation on job state transitions

### Anti-Patterns to Avoid

- **Variable expansion in heredocs:** Always use single-quoted heredocs (`<<'EOF'`) for code generation
- **Assuming file existence:** Always check `exists()` or use `missing_ok=True` before destructive operations
- **Default-to-failure for ambiguous states:** Use retry logic for transient conditions
- **Silent truncation:** When reporting mismatches, include total count alongside sample (see Issue #4)

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Shell variable escaping | Custom quoting logic | `printf '%q'` or single-quoted heredocs | Handles all edge cases (backticks, $(), etc.) |
| Retry with backoff | Custom sleep loops | `tenacity` library or exponential backoff pattern | Handles jitter, max retries, failure modes |
| File locking | Custom lock files | `fcntl.flock()` or `fasteners` library | POSIX-compliant, handles process crashes |
| Subprocess output streaming | In-memory buffers | Stream to temp files or use `subprocess.PIPE` with threads | Prevents OOM on large outputs |
| Atom index validation | First-N reporting | Full count + sample + detailed log file | Debugging large molecules requires full data |

**Key insight:** Shell injection and race conditions have well-documented solutions. Reinventing these risks introducing new vulnerabilities.

## Common Pitfalls

### Pitfall 1: Unquoted Variable Expansion in Generated Scripts

**What goes wrong:** Shell metacharacters in file paths (backticks, `$()`, spaces) get interpreted during script execution, leading to command injection or parsing errors.

**Why it happens:** Developer assumes file paths are "safe" alphanumeric strings, but scientific workflows often have:
- Ligand names with parentheses: `compound_1(2).mol2`
- Generated paths with shell variables: `$SCRATCH/ligands`
- Spaces from user-provided inputs

**How to avoid:**
1. Use `printf '%q'` to escape shell variables before heredoc creation
2. Use single-quoted heredocs (`<<'EOF'`) and sed-based substitution
3. Never use unquoted `${var}` in `<<EOF` heredocs for user-controlled data

**Warning signs:**
- ShellCheck warnings SC2086, SC2248
- Job failures with "command not found" for paths containing spaces
- Unexplained variable expansion in SBATCH output

**Source:** A_SECURITY_PERFORMANCE.md Issue #1 (Critical)

---

### Pitfall 2: Race Conditions in Slurm Status Checking

**What goes wrong:** Job completes between `squeue` (active jobs) and `sacct` (historical jobs) queries, causing workflow to incorrectly report failure.

**Why it happens:** Slurm's `squeue` and `sacct` use different data sources:
- `squeue` queries controller memory (real-time, short retention)
- `sacct` queries accounting database (persisted, replication lag)

Short-lived jobs can disappear from `squeue` before `sacct` replication completes.

**How to avoid:**
1. Implement retry logic with exponential backoff for UNKNOWN states
2. Query `sacct` multiple times before declaring failure
3. Use job output files as ground truth when status is ambiguous
4. Set reasonable timeout (e.g., 5 minutes) for status propagation

**Warning signs:**
- Jobs that completed successfully but workflow reports FAILED
- Intermittent failures that resolve on re-run
- Log messages showing UNKNOWN status immediately after job completion

**Source:** A_SECURITY_PERFORMANCE.md Issue #3 (Critical), Slurm documentation on job state visibility

---

### Pitfall 3: Insufficient Error Context in Truncated Logs

**What goes wrong:** Reporting only first N items of a mismatch hides the true extent of the problem, making debugging difficult or impossible.

**Why it happens:** Developer tries to keep error messages concise for readability, but loses critical diagnostic information.

**How to avoid:**
1. Report total count alongside sample: `"Missing in ITP (457 total): [first 10]..."`
2. Save full lists to separate diagnostic file
3. Reference full diagnostic file in error message
4. Use structured logging to separate user-facing summary from debug details

**Warning signs:**
- Error message says "...and more" without indicating how many more
- User unable to fix issue without re-running with modified error reporting
- Debugging requires code changes to see full data

**Source:** A_SECURITY_PERFORMANCE.md Issue #4 (Critical)

---

### Pitfall 4: Memory Exhaustion from Subprocess Output Buffering

**What goes wrong:** Long-running commands (GROMACS mdrun, MM/PBSA) generate gigabytes of log output, which `subprocess.run(capture_output=True)` buffers entirely in memory, causing OOM kills.

**Why it happens:** Python's `subprocess.run()` with `capture_output=True` reads all stdout/stderr into memory before returning. GROMACS verbosity can produce 100GB+ logs for production MD.

**How to avoid:**
1. Stream output to files using `stdout=file_handle`
2. Read last N lines from log file for error reporting
3. Use `subprocess.Popen()` with manual stream handling for real-time monitoring
4. Configure tools to reduce verbosity when full logs aren't needed

**Warning signs:**
- OOM killer messages in system logs
- Jobs that succeed when run manually but fail in wrapper
- Memory usage proportional to job runtime

**Status:** Addressed in Plan 06.1-03 (Task 4) - scripts/infra/executor.py and scripts/agents/runner.py currently use capture_output=True and require conversion to file streaming.

**Source:** A_SECURITY_PERFORMANCE.md Issue #11 (High)

---

### Pitfall 5: Default Timeout None Enabling Infinite Loops

**What goes wrong:** Job gets stuck in non-terminal state (e.g., suspended, held by scheduler policy), and `while True` loop with `timeout=None` runs forever.

**Why it happens:** Developer assumes all jobs eventually reach terminal state, but scheduler policies can indefinitely suspend jobs (quota, maintenance, fair-share).

**How to avoid:**
1. Always set a default timeout (e.g., 7 days for production MD)
2. Document expected runtimes and set timeout to 2-3× expected
3. Log elapsed time at each poll interval
4. Provide manual override mechanism for exceptionally long jobs

**Warning signs:**
- Workflow appears "hung" with no progress or error
- High CPU usage from tight polling loop
- No automated recovery mechanism

**Source:** A_SECURITY_PERFORMANCE.md Issue #5 (High)

### Defensive Cache Clearing (Full Implementation)

```python
# Source: Recommended fix for conversion_cache.py:73-94
def clear(self, source_path: Path | None = None) -> None:
    """Clear cache entries with defensive error handling."""
    import logging
    logger = logging.getLogger(__name__)
    
    if source_path is None:
        # Clear all cache files
        for path in self.cache_dir.glob("*"):
            if path.is_file():
                try:
                    path.unlink(missing_ok=True)
                except OSError as e:
                    logger.warning(f"Failed to delete cache file {path}: {e}")
                    continue
        return
    
    # Clear specific source's cache
    source_str = str(source_path.resolve())
    cleared_count = 0
    error_count = 0
    
    for meta_file in self.cache_dir.glob("*.meta"):
        try:
            meta = json.loads(meta_file.read_text(encoding="utf-8"))
        except (OSError, json.JSONDecodeError) as e:
            logger.warning(f"Failed to read metadata {meta_file}: {e}")
            continue
        
        if meta.get("source_path") != source_str:
            continue
        
        # Delete both result and metadata files
        result_file = meta_file.with_suffix(".result")
        try:
            if result_file.exists():
                result_file.unlink(missing_ok=True)
            if meta_file.exists():
                meta_file.unlink(missing_ok=True)
            cleared_count += 1
        except OSError as e:
            logger.warning(f"Failed to clear cache for {source_path}: {e}")
            error_count += 1
    
    logger.info(f"Cleared {cleared_count} cache entries for {source_path} ({error_count} errors)")
```

### Retry Logic for Job Status (Full Implementation)

```python
# Source: Recommended fix for executor.py:398-410
def wait_for_job(
    self,
    job_id: str,
    poll_interval: int = 30,
    timeout: int | None = None
) -> tuple[str, str]:
    """Wait for job completion with retry logic for transient states."""
    import logging
    logger = logging.getLogger(__name__)
    
    # Default timeout to prevent infinite waits
    effective_timeout = timeout or (7 * 24 * 3600)  # 7 days default
    start_time = time.time()
    
    # Retry configuration for UNKNOWN state
    max_unknown_retries = 3
    unknown_backoff_base = poll_interval
    
    while True:
        elapsed = time.time() - start_time
        if elapsed > effective_timeout:
            raise TimeoutError(
                f"Timeout waiting for job {job_id} after {effective_timeout}s "
                f"(elapsed: {elapsed:.1f}s)"
            )
        
        # Get current job status
        status = self.get_job_status(job_id)
        
        # Terminal states - return immediately
        if status in self.TERMINAL_STATES:
            return (status, self.get_output_file(job_id))
        
        # Unknown state - retry with backoff
        if status == 'UNKNOWN':
            logger.debug(f"Job {job_id} in UNKNOWN state, checking sacct...")
            
            for attempt in range(max_unknown_retries):
                terminal_state = self.get_job_terminal_state(job_id)
                
                if terminal_state == 'COMPLETED':
                    return ('COMPLETED', self.get_output_file(job_id))
                
                if terminal_state in self.FAILURE_STATES or terminal_state in self.TERMINAL_STATES:
                    return (terminal_state, self.get_output_file(job_id))
                
                if terminal_state != 'UNKNOWN':
                    # Found definitive state, return it
                    return (terminal_state, self.get_output_file(job_id))
                
                # Still unknown, wait before retry
                if attempt < max_unknown_retries - 1:
                    backoff_time = unknown_backoff_base * (2 ** attempt)
                    logger.debug(
                        f"Job {job_id} state still unknown, "
                        f"retry {attempt+1}/{max_unknown_retries} in {backoff_time}s"
                    )
                    time.sleep(backoff_time)
            
            # Exhausted retries, fail with context
            logger.error(
                f"Job {job_id} in indeterminate state after {max_unknown_retries} retries. "
                f"Check scheduler status and job output manually."
            )
            return ('FAILED', self.get_output_file(job_id))
        
        # Non-terminal, known state - continue polling
        time.sleep(poll_interval)
```
